# P23: S2S Native Audio Model Voice Quality Limits

## Pattern

Speech-to-speech (S2S) native audio models generate voice output with quality issues that cannot be fixed through configuration or prompting. The model's audio generation layer has hard limits: accent drift in non-English languages, mid-sentence truncation (premature turn completion), and no speech rate/pitch/volume controls. These are server-side limitations, not client-side bugs.

This pattern is distinct from P19 (prompt hard limits) because the issue is in the audio generation layer specifically, not the text generation layer. The model may "understand" instructions about speech quality but its audio synthesis pipeline does not implement them.

## Check List (30-Second Diagnosis)

- [ ] Is the model a native-audio S2S model (not a text model + separate TTS)?
- [ ] Does the issue affect audio quality specifically (accent, truncation, pacing) rather than content?
- [ ] Have you confirmed the issue persists across 10+ calls with different inputs?
- [ ] Does the same content sound correct when generated by a standalone TTS engine?

If 3+ checks are "yes," this pattern likely matches. Stop tuning the S2S model and evaluate cascade architecture.

## Examples

### Example 1: German Accent on Gemini Live (~50% of calls)
**Setup:** `gemini-3.1-flash-live-preview` with voice "Aoede", language "de", temperature 1.0. System prompt entirely in German with explicit "Sprich langsam und deutlich" directives.
**Symptom:** ~50% of calls have English-accented German. Accent appears randomly, sometimes mid-conversation. Community reports confirm accent switching spontaneously (Google Support thread #383995287).
**Root cause:** Native-audio models auto-detect language. No `language_code` parameter is supported for native-audio variants. Voices trained primarily on English data. System prompt language directives are processed by text layer but have limited effect on audio synthesis layer.
**Fix:** (1) Quick: test "Clear"/"Firm" voices (Kore, Charon, Erinome, Iapetus) -- less casual voices may produce clearer German diction. (2) Medium: add explicit German enforcement to system prompt ("DU ANTWORTEST AUSSCHLIESSLICH AUF DEUTSCH"). (3) Long-term: switch to cascade architecture with a dedicated German TTS (ElevenLabs, Cartesia).

### Example 2: Word Swallowing / Mid-Sentence Truncation (Gemini Live)
**Setup:** Any Gemini Live native-audio model. Conversation with 5+ turns or tool calls.
**Symptom:** Model stops speaking mid-sentence (sometimes mid-word). Premature `turnComplete` message arrives without `interrupted: true`. Remaining audio never delivered. Frequency increases over session duration.
**Root cause:** Server-side bug in Gemini Live audio generation (GitHub googleapis/python-genai #2117, open 8+ months, P2 priority, ~40 developer confirmations). Aggravating factors: non-English languages, tool calls/function returns, growing conversation context, `enable_affective_dialog`. A developer implemented 5 layers of client-side protection (hardware AEC, echo gating, VAD confirmation, non-interruptible mode, disabled automatic activity detection) -- truncation persists, confirming server-side root cause.
**Fix:** No client-side fix exists. Mitigations: (1) SilenceWatchdog to detect and recover from truncated output. (2) Keep conversations short (reduces frequency). (3) Switch to cascade architecture to eliminate the S2S audio path entirely.

### Example 3: No Speech Rate/Pitch Controls
**Setup:** Gemini Live S2S mode. Want to slow down speech for telephone clarity.
**Symptom:** `speaking_rate`, `pitch`, `volume_gain_db` parameters are NOT available in the Live API. Feature request (GitHub #2029) is P3 priority.
**Root cause:** API design gap. The standalone TTS API (`gemini-3.1-flash-tts-preview`) supports these controls plus 200+ inline audio tags. The Live API does not.
**Fix:** Prompt-level "speak slowly and clearly" has some effect but is unreliable. For guaranteed control, use cascade architecture with standalone TTS.

## Fix Strategy

### Tier 1: Quick Mitigations (minutes)
1. **Voice selection** -- Test all 30 voices systematically. "Firm" and "Clear" voices (Kore, Charon, Erinome, Iapetus, Alnilam) tend to produce clearer non-English speech than "Breezy" or "Excitable" voices
2. **Temperature** -- Must be 1.0 for Gemini 3.1. Lower temperatures cause loops AND degrade speech quality
3. **System prompt** -- Add explicit language enforcement at prompt END (where constraints are best attended): "DU ANTWORTEST AUSSCHLIESSLICH AUF DEUTSCH. Artikuliere jedes Wort vollstaendig."
4. **SilenceWatchdog** -- Detect truncated output and nudge the model to continue speaking

### Tier 2: Architecture Change (days)
5. **Cascade architecture**: Replace S2S with Deepgram STT + Gemini 2.5 Flash (text LLM) + ElevenLabs/Cartesia TTS
   - Full control over each component
   - Latency: ~400-600ms total (potentially faster than Gemini Live S2S at ~2.98s TTFT)
   - Voice quality: ElevenLabs MOS 4.72, Cartesia ~90ms TTFA
   - Trade-off: loses S2S's natural turn-taking; must implement VAD + interruption handling

### Tier 3: Platform Switch (weeks)
6. **Alternative S2S**: OpenAI Realtime API, or wait for Gemini Live fixes

## Prevention

- **Do not promise voice quality parity** with standalone TTS when using S2S native audio models
- **Benchmark S2S voice quality** on target language BEFORE committing to architecture
- **Design for cascade fallback** -- keep STT/LLM/TTS as separate services that can be swapped in
- **Monitor server-side bug trackers** (googleapis/python-genai, google-gemini/live-api-web-console) for fix releases
- When evaluating any S2S model: test with 20+ calls in the target language before production commitment

## Debugger Strategy

S2S audio quality issues are not debuggable with traditional breakpoints. Use these diagnostic approaches instead.

**Audio Capture:**
- Record raw PCM output from the S2S model BEFORE any resampling or codec conversion
- Compare with the same text rendered through standalone TTS
- If raw S2S audio has the quality issue, it is model-side (this pattern). If raw audio is clean but phone audio is degraded, investigate resampling/codec path

**Breakpoints:**
- `BotStoppedSpeakingFrame` handler -- Check if this fires mid-sentence (word swallowing). Compare `_tool_in_progress` counter to rule out tool-related truncation
- `turnComplete` WebSocket message handler -- Check `interrupted` field. If `false` and speech was incomplete, this confirms the server-side truncation bug

**Watch Expressions:**
- `gemini_live._sample_rate` -- Should be 24000. Mismatch causes pitch/speed artifacts
- `transport.params.audio_out_sample_rate` -- Should be 8000 for Twilio. Mismatch causes aliasing

**Isolation Technique:**
Play the same German text through (1) Gemini Live S2S, (2) `gemini-3.1-flash-tts-preview` standalone, (3) ElevenLabs. If only (1) has quality issues, this pattern is confirmed.

**Expected Evidence:**
Confirms pattern: standalone TTS produces clean German, S2S produces accented German with same text content. Rules it out: all three engines produce similar quality -- issue is in audio transport/resampling, not the model.
